# Topic 0.1 — Modern C++ I: types, pointers, references, stack vs heap

**Exercise statements.** Designed for this topic, in increasing difficulty. Write your own code; the proof of understanding is compiling code in Git, not notes. Solutions are **not** included here.

## Exercise 1 — Types and value semantics (warm-up)

1. Print, with `sizeof`, the size of: `int`, `char`, `bool`, `double`, `int*`, `double*`, and a small `struct` of your own (e.g. one `int` + one `char`).
2. In the README, comment on:
   - What do `int*` and `double*` have in common?
   - Why is the struct's size not exactly the sum of its members? *(hint: alignment/padding — covered in depth in 0.4; here just observe.)*
3. Write `void addOne(int n)` that does `n += 1` inside. Call it from `main`, printing the variable before and after, to show that the caller's variable does **not** change. Explain why in one line.

## Exercise 2 — Address-of and dereference

- Declare `int x = 10;` and `int* p = &x;`
- Print: the value of `x`, its address (`&x`), the value held by `p`, the address where `p` lives (`&p`), and `*p`.
- Change `x` to 99 **through `p`** (not by touching `x` directly). Print `x` to confirm.
- Print `sizeof(p)` and `sizeof(*p)`. In the README, explain the difference between those two numbers.

## Exercise 3 — swap by value vs pointer vs reference (the classic)

- Write three functions: `swapByValue(int a, int b)`, `swapByPointer(int* a, int* b)`, and `swapByReference(int& a, int& b)`.
- From `main`, with two variables, call all three and check (by printing) what happens to the caller's variables in each case.
- In the README: why does `swapByValue` not work? What is the practical difference between the pointer version and the reference version (call syntax, possibility of null)?

## Exercise 4 — Lifetime: stack vs heap (and the footgun)

- Write `int* makeOnHeap(int v)` that returns `new int(v)`. In `main`: use it, print `*result`, free it with `delete`, and set the pointer to `nullptr`.
- Show that the heap object outlives the function that created it (unlike a stack variable).
- In a comment, explain why the following is wrong (dangling pointer). Do **not** dereference it; reason about it in writing:

```cpp
int* makeOnStack(int v) { int local = v; return &local; }
```

- **Challenge:** cause a memory leak on purpose (allocate with `new`, don't free), comment on it, then fix it. (You won't "see" the leak until 0.10 with sanitizers; reasoning about it is enough here.)

## Exercise 5 — Integrator: a manual dynamic buffer (no `std::vector`)

- Define `struct IntBuffer { int* data; size_t size; };`
- Write `IntBuffer makeBuffer(size_t n)`: allocate with `new int[n]`, fill it (e.g. with `i*i`), and return the struct.
- Write `int sumBuffer(const IntBuffer& buf)`: iterate and sum — note that you pass it by const reference to avoid copying.
- Write `void freeBuffer(IntBuffer& buf)`: `delete[] buf.data;` then set `buf.data = nullptr;` and `buf.size = 0;`
- In `main`: create, sum, print, free.
- In the README, reason about this: what would happen if you **copied** an `IntBuffer` (two structs sharing the same `data` pointer) and both called `freeBuffer`? This is exactly the problem that RAII and the rule of three/five solve in 0.2 — you are building, on purpose, the problem the next topic fixes.
