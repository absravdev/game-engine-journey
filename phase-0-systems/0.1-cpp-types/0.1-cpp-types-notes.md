# 0.1 — Modern C++ I: types, pointers, references, stack vs heap

Working code and observations for topic 0.1.

## Exercise 1 — Types and value semantics

Source: `ex1.cpp`.

### 1.1 — sizeof of types, pointers, and a struct
Prints the size in bytes of `int`, `char`, `bool`, `double`, `int*`, `double*`,
and a small `struct { int x; char c; }`.

### 1.2 — Observations
- **What `int*` and `double*` have in common:** both are pointers, that is,
  memory addresses, and every memory address takes the same space (8 bytes on
  64-bit, 4 bytes on 32-bit), regardless of what they point to.
- **Why the struct's size is not exactly the sum of its members:** members are
  stored in blocks; the total size is rounded up to a multiple of the alignment,
  which determines the block size.

### 1.3 — Value semantics (pass by value)
`addOne(int n)` does `n += 1` and returns nothing. Calling `addOne(a)` does not
change `a`, because inside the function `n` is not `a` itself but a copy of it;
the function only modifies its copy.

## Exercise 2 — Address-of and dereference

Source: `ex2.cpp`.

### 2.1 — Setup
`int x = 10;` then `int* p = &x;` — `p` is a pointer whose value is the address of `x`.

### 2.2 — Printing addresses and values
Prints `x`, `&x`, `p`, `&p`, `*p`. The non-obvious relationships, and why:

- `x` → the **value** stored in x (`10`).
- `&x` → the **address** where x lives (`&` in an expression = address-of).
- `p` → the **value** of p, which is the address of x. `p == &x` is not a
  coincidence: it is exactly what `p = &x` stored.
- `&p` → the address of the box `p` itself. Different from `&x` because `p` is a
  separate variable living in its own memory slot.
- `*p` → **dereference**: go to the address held in p and read what is there →
  x's value (`10`). So `*p == x`.

The single idea behind all of it: **a pointer is a normal variable that lives in
its own slot and whose content is an address.** There are two different addresses
in play at once (x's and p's), which is what makes this confusing at first.

The concrete addresses change on every run (it is the stack — a different slot
each time); the **relationships** never change: `&x == p` and `*p == x` always hold.

### 2.3 — Modifying through the pointer
`*p = 20;` writes `20` into the box at the address p holds — and that box is `x`.
So `x` becomes `20` without naming `x` directly. This is the proof that dereference
reaches the real object, not a copy.

### 2.4 — sizeof(p) vs sizeof(*p)  (the part I got backwards)
- `sizeof(p)` = **8** — `p` is a pointer, i.e. an address; all addresses occupy
  8 bytes on a 64-bit build, regardless of the pointed-to type.
- `sizeof(*p)` = **4** — `*p` is the `int` p points to; `sizeof(int)` is 4.

I first had this reversed (thought `p` was the int and `*p` was the address). The
program is the arbiter: it prints `8` then `4` — the opposite of that wrong guess.

General rule:
- `sizeof(<pointer>)` → always the size of an address (8 on 64-bit), whatever it points to.
- `sizeof(*<pointer>)` → the size of the pointed-to type.

This is the **same fact** as Exercise 1 ("`int*` and `double*` take the same
space"): a pointer is an address, so its size is the address size.

### Memory diagram

```
          +--------------------------+
   x  --> |  address : 0x...9B4      |   = &x
          |  value   : 10            |   =  x   (and *p)
          +--------------------------+
                      ^
                      |  p holds this address -> "points to x"
                      |
          +--------------------------+
   p  --> |  address : 0x...9D8      |   = &p
          |  value   : 0x...9B4      |   =  p
          +--------------------------+
```

## Exercise 3 — swap by value vs pointer vs reference

Source: `ex3.cpp`.

The whole point: write the same swap three ways and observe which ones reach the
caller's variables. value = copies (no), pointer = addresses (yes), reference =
aliases (yes). Reset `a = 1; b = 2;` before each call so all three start equal.

### 3.1 — Mistakes I made (the valuable part)

1. **Broken swap: `a = b; b = a;`.** Assignment overwrites, so the first line
   already destroys `a`'s original value; the result is `2, 2` instead of `2, 1`.
   A swap needs a THIRD variable to hold the value about to be overwritten:
   ```cpp
   int temp = a;
   a = b;
   b = temp;
   ```

2. **swapByPointer: I assigned the pointers, not the values.** I wrote `a = b;`
   inside, which only repoints the local pointer copy and never touches any int.
   To change the originals you must dereference: `int temp = *a; *a = *b; *b = temp;`.

3. **Forgot the reset between calls.** Since pointer/reference versions really swap,
   without resetting `a, b` the reference call swaps back what the pointer call
   already swapped, and the output looks like reference "did nothing".

4. **Called the value params "references".** In `swapByValue` the params are
   COPIES, not references — and that is exactly why it does not reach main. Copy and
   reference are opposites: a copy is a new independent box; a reference is the same
   box under another name.

### 3.2 — Key ideas
- **Copy / address / alias:** value makes new boxes (copies); pointer holds the
  addresses of main's boxes and reaches them through `*a`/`*b`; reference's params
  ARE main's boxes under a second name.
- **A pointer is passed by value too.** The function receives a COPY of the address.
  Reassigning the pointer (`a = b`) never reaches main; only writing through it
  (`*a = ...`) does. "You can modify the pointee, not rebind the caller's pointer."
- **pointer vs reference:** pointer is explicit — call `swapByPointer(&a, &b)`, use
  `*a`/`*b` inside; can be null; can be repointed. Reference is implicit — call
  `swapByReference(a, b)`, use `a`/`b` inside; cannot be null; cannot be rebound.
  The reference call looks IDENTICAL to a value call — a famous gotcha (you cannot
  tell from the call site that the function can modify your variables).

## Exercise 4 — Lifetime: stack vs heap (and the footguns)

Source: `ex4.cpp`.

Heap objects outlive the function that created them, but their lifetime is MANUAL:
allocate with `new`, free with `delete`. Stack objects have AUTOMATIC lifetime —
they die when their scope ends.

### 4.1 — Mistakes / footguns I hit (the valuable part)

1. **Discarded the return value.** `makeOnHeap(a);` without capturing the returned
   pointer leaks immediately: the heap int is allocated and its only address is
   thrown away, so it can never be deleted. Capture it: `int* result = makeOnHeap(a);`.

2. **Printed `result` instead of `*result`.** `result` is the address; `*result`
   is the heap value. Seeing `*result == 1` AFTER the function returned is the proof
   the heap object outlived it.

3. **Order matters: dereference BEFORE delete.** After `delete result;` the memory
   is freed; reading `*result` then is use-after-free (UB). Order: `*result` → delete
   → nullptr.

4. **Forgot `result = nullptr;`.** `delete` frees the memory but does NOT change the
   pointer — it still holds the old address, now dangling. nullptr makes it clearly
   "points to nothing".

5. **`new int b = 5;` does not compile.** `new` is an operator, not a declaration
   modifier. It allocates an ANONYMOUS heap object and returns its address. Correct:
   `int* b = new int(5);` — the name `b` belongs to the pointer, not to the heap int
   (which never gets a name; you reach it only through `*b`).

6. **English: "memory direction" → "memory address".** Recurring false friend.

### 4.2 — Dangling pointer (makeOnStack)
`int* makeOnStack(int v) { int local = v; return &local; }` is broken: `local` is a
stack variable destroyed when the function returns, so the returned address points
to dead / reused memory = a **dangling pointer**; dereferencing it is undefined
behavior. The compiler catches this one (`-Wreturn-local-addr`). To return a value
safely, return BY VALUE (`return local;`) — the value is copied out before the frame
dies. Use the heap only when the object must deliberately outlive the function.

### 4.3 — Memory leak (demo + fix)
A leak = heap memory you never free. In the demo, reassigning the only pointer
(`b = &c;`) loses the heap int's address → it can never be deleted → leaked. Fix:
`delete` the heap int WHILE you still hold its address, before reassigning. Footgun:
never `delete` a pointer that holds a STACK address (`&c`) — deleting something not
returned by `new` is UB.

Observed concretely: after `delete result`, the next `new int(5)` reused the exact
same address. So the freed slot was recycled — a dangling pointer's address can end
up owned by a different object. That is why dangling pointers are dangerous, and why
you set the pointer to nullptr after delete.

AddressSanitizer (`-fsanitize=address`, a 0.10 tool) confirmed the reasoning: it
reported exactly the leaked allocation (4 bytes, 1 object) and ignored the freed one.

> Forward link: having to remember `delete` by hand is exactly the pain RAII and
> smart pointers remove in **0.2**. Naive before abstraction — feel it first.

## Exercise 5 — Integrator: manual dynamic buffer (no `std::vector`)

Source: `ex5.cpp`.

Build a minimal `std::vector<int>` by hand to feel manual heap management:
`struct IntBuffer { int* data; size_t size; };` plus three functions — `makeBuffer`
(allocates with `new int[n]`, fills with `i*i`, returns the struct), `sumBuffer`
(reads via `const IntBuffer&`, sums), and `freeBuffer` (releases with `delete[]`,
then nulls `data` and zeroes `size`).

Key idea: **`IntBuffer` is a handle, not the data.** The struct holds only a pointer
and a size; the array itself lives separately on the heap. That is why returning the
struct by value is cheap — it copies the handle, not the array — and why `new int[n]`
(heap) is required: `n` is a runtime value, so the array cannot be a fixed-size stack
array.

### 5.1 — Copying an `IntBuffer`: the double free (the notes question)

What happens with:

```cpp
IntBuffer a = makeBuffer(5);
IntBuffer b = a;          // copy the struct
freeBuffer(a);
freeBuffer(b);
```

1. **What is copied.** `IntBuffer b = a;` copies the **handle** (the `data` pointer
   and the `size`), NOT the array. This is a **shallow copy**.
2. **Why both point to the same array.** `data` is a pointer, i.e. an address.
   Copying the struct copies that address, so `a.data` and `b.data` hold the **same
   address** → one array on the heap, two owners. (Same fact as ex3: a pointer is
   copied by value — you copy the address, not the pointee.)
3. **Why the double free is UB.** `freeBuffer(a)` runs `delete[]` and the shared
   array is gone; it also sets `a.data = nullptr`, so `a` ends up safe. But it does
   NOT touch `b`: `b.data` still holds the freed address and `b.size` still says `5`,
   so `b` is now a **dangling** handle. Then `freeBuffer(b)` runs `delete[]` on
   already-freed memory — a **double free**, which is **undefined behavior** (heap
   corruption, typically a crash).

The deeper point: once the copy exists there is **no good manual move**. Freeing `b`
is a double free; *not* freeing it leaves a dangling pointer whose use is
use-after-free (ex4). The damage is born at `IntBuffer b = a;`, not at the `delete[]`
— an innocent copy broke the "one array, one owner" invariant.

> Forward link: this is exactly what RAII and the rule of three/five fix in **0.2** —
> make the type define what copying means (deep copy, or forbid the copy) and free
> memory automatically when the owner dies. Naive before abstraction — feel it first.

## New concepts

- **bookkeeping** — the work the allocator does to find a free block of memory when you allocate on the heap (e.g. with `new`).
- **padding** — filler bytes the compiler inserts between or after struct members so that each member starts at a properly aligned address (and the struct's total size is a multiple of its alignment), which lets the CPU read the members efficiently. Where the padding lands depends on the order of the members:

```
struct MyStruct {
    int x;
    char c;
};

offset:  0    1    2    3    4    5    6    7
        [------- int x -------][ c ][-- padding --]
```

```
struct MyStruct2 {
    char c;
    int x;
};

offset:  0    1    2    3    4    5    6    7
        [ c ][--- padding ---][------- int x -------]
```

Same total size (8 bytes), but the padding moves: at the end (after `c`) in the first; in the middle in the second, so that `int x` still starts at a 4-aligned offset.

- **pointer size** — `int*` and `double*` are memory addresses; on a 64-bit build an address takes 8 bytes, on 32-bit it takes 4 — regardless of what they point to.
- **address-of (`&`)** — in an expression, `&x` yields the address of `x`.
- **dereference (`*`)** — in an expression, `*p` follows the pointer to the object it points to, to read or write that object.
- **`&` and `*` are context-dependent** — in a *declaration*, `int* p` means "pointer type" and `int& r` means "reference type"; in an *expression*, `&` is address-of and `*` is dereference. Same symbols, different jobs.
- **alias (reference)** — a reference is not a copy and not an address; it is a second name for an object that already exists. Same box, two labels.
- **stack frame** — the slice of stack memory given to a function call for its locals; destroyed when the function returns.
- **automatic vs manual lifetime** — stack objects live until their scope ends (automatic, managed for you); heap objects (`new`) live until you `delete` them (manual, your responsibility).
- **`new` / `delete`** — `new` allocates an anonymous object on the heap and returns its address; `delete` frees it. Every `new` needs exactly one matching `delete`.
- **dangling pointer** — a pointer to memory that is no longer valid (a freed heap object, or a stack variable whose function returned). Dereferencing it is UB.
- **memory leak** — heap memory that is never freed because its last pointer was lost or `delete` was never called. Harmless in a tiny program (the OS reclaims on exit), fatal in a long-running one (e.g. a game engine).
- **use-after-free** — reading or writing through a pointer after the object it points to has been freed; a form of dangling-pointer UB.
