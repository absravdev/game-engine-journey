# 0.1 — Modern C++ I: types, pointers, references, stack vs heap

Working code and observations for topic 0.1.

## Exercise 1 — Types and value semantics

Source: `ex1.cpp`.

### 1.1 — sizeof of types, pointers, and a struct
Prints the size in bytes of `int`, `char`, `bool`, `double`, `int*`, `double*`,
and a small `struct { int x; char c; }`.

### 1.2 — Observations
- **What `int*` and `double*` have in common:** both are pointers, that is,
  memory addresses, and every memory address takes the same space (8 bytes on
  64-bit, 4 bytes on 32-bit), regardless of what they point to.
- **Why the struct's size is not exactly the sum of its members:** members are
  stored in blocks; the total size is rounded up to a multiple of the alignment,
  which determines the block size.

### 1.3 — Value semantics (pass by value)
`addOne(int n)` does `n += 1` and returns nothing. Calling `addOne(a)` does not
change `a`, because inside the function `n` is not `a` itself but a copy of it;
the function only modifies its copy.

## Exercise 2 — Address-of and dereference

Source: `ex2.cpp`.

### 2.1 — Setup
`int x = 10;` then `int* p = &x;` — `p` is a pointer whose value is the address of `x`.

### 2.2 — Printing addresses and values
Prints `x`, `&x`, `p`, `&p`, `*p`. The non-obvious relationships, and why:

- `x` → the **value** stored in x (`10`).
- `&x` → the **address** where x lives (`&` in an expression = address-of).
- `p` → the **value** of p, which is the address of x. `p == &x` is not a
  coincidence: it is exactly what `p = &x` stored.
- `&p` → the address of the box `p` itself. Different from `&x` because `p` is a
  separate variable living in its own memory slot.
- `*p` → **dereference**: go to the address held in p and read what is there →
  x's value (`10`). So `*p == x`.

The single idea behind all of it: **a pointer is a normal variable that lives in
its own slot and whose content is an address.** There are two different addresses
in play at once (x's and p's), which is what makes this confusing at first.

The concrete addresses change on every run (it is the stack — a different slot
each time); the **relationships** never change: `&x == p` and `*p == x` always hold.

### 2.3 — Modifying through the pointer
`*p = 20;` writes `20` into the box at the address p holds — and that box is `x`.
So `x` becomes `20` without naming `x` directly. This is the proof that dereference
reaches the real object, not a copy.

### 2.4 — sizeof(p) vs sizeof(*p)  (the part I got backwards)
- `sizeof(p)` = **8** — `p` is a pointer, i.e. an address; all addresses occupy
  8 bytes on a 64-bit build, regardless of the pointed-to type.
- `sizeof(*p)` = **4** — `*p` is the `int` p points to; `sizeof(int)` is 4.

I first had this reversed (thought `p` was the int and `*p` was the address). The
program is the arbiter: it prints `8` then `4` — the opposite of that wrong guess.

General rule:
- `sizeof(<pointer>)` → always the size of an address (8 on 64-bit), whatever it points to.
- `sizeof(*<pointer>)` → the size of the pointed-to type.

This is the **same fact** as Exercise 1 ("`int*` and `double*` take the same
space"): a pointer is an address, so its size is the address size.

## New concepts

- **bookkeeping** — the work the allocator does to find a free block of memory when you allocate on the heap (e.g. with `new`).
- **padding** — filler bytes the compiler inserts between or after struct members so that each member starts at a properly aligned address (and the struct's total size is a multiple of its alignment), which lets the CPU read the members efficiently. Where the padding lands depends on the order of the members:

```
struct MyStruct {
    int x;
    char c;
};

offset:  0    1    2    3    4    5    6    7
        [------- int x -------][ c ][-- padding --]
```

```
struct MyStruct2 {
    char c;
    int x;
};

offset:  0    1    2    3    4    5    6    7
        [ c ][--- padding ---][------- int x -------]
```

Same total size (8 bytes), but the padding moves: at the end (after `c`) in the first; in the middle in the second, so that `int x` still starts at a 4-aligned offset.

- **pointer size** — `int*` and `double*` are memory addresses; on a 64-bit build an address takes 8 bytes, on 32-bit it takes 4 — regardless of what they point to.
- **address-of (`&`)** — in an expression, `&x` yields the address of `x`.
- **dereference (`*`)** — in an expression, `*p` follows the pointer to the object it points to, to read or write that object.
- **`&` and `*` are context-dependent** — in a *declaration*, `int* p` means "pointer type" and `int& r` means "reference type"; in an *expression*, `&` is address-of and `*` is dereference. Same symbols, different jobs.
